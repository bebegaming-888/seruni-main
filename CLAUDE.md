# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Stack

- **Framework**: TanStack Start (TanStack Router + React Start) with Vite
- **UI**: React 19, Tailwind CSS v4 (CSS-based), Radix UI primitives, shadcn/ui-style components
- **Forms**: React Hook Form + Zod
- **Charts**: Recharts
- **PDF**: pdf-lib + qrcode (generated in edge functions, server-side)
- **Backend**: Supabase (Auth + PostgreSQL + RLS), Cloudflare Pages Functions (Edge)
- **Deployment**: Cloudflare Pages

> **Vite config is auto-generated by `@lovable.dev/vite-tanstack-config`** — do NOT edit `vite.config.ts` manually. Adding `@rollup/plugin-*`, `tailwindcss`, `vite-plugin-*`, etc. will break the build. Full stack details, roadmap, execution principles, and feature priorities are in **`docs/GOALS.md`** — always check there first.

## Architecture

### Routing

TanStack Router with file-based routes in `src/routes/`. Route tree is auto-generated to `routeTree.gen.ts` — **do not edit manually**. TanStack Query handles async state.

`src/router.tsx` exports `getRouter()` with a custom `DefaultErrorComponent` — global error/404 page. Dev mode shows error details; production shows a clean fallback.

### Auth: Two Systems

- **Admin Auth** (`src/lib/auth.ts`): localStorage/sessionStorage-based. Superadmin is always sourced from `VITE_ADMIN_USER`/`VITE_ADMIN_PASS` — **cannot be removed or renamed**. Additional users stored in localStorage, managed via Settings.
- **Warga Auth** (`src/lib/warga-auth.ts`): login with NIK + WhatsApp OTP. Session in `sessionStorage` (tab lifetime). Dev mode: OTP "123456" works without Supabase.
  - Edge functions: `functions/api/auth/request-otp.ts`, `functions/api/auth/verify-otp.ts`
  - DB migration: `002_warga_auth.sql` (`otp_requests`, `warga_sessions`, `warga_trackings`)
  - Protected route: `/masuk/warga`
- **Supabase Auth**: schema exists in migrations but not wired in the frontend. Supabase RLS uses the `admin_users` table, not built-in auth.

> `src/lib/supabase.ts` returns `null` when env vars are absent. Always check `isSupabaseConfigured` or null-check — never assume the client always exists.

### E-Surat Workflow

5-status pipeline: `Menunggu Verifikasi → Diverifikasi → Menunggu Approval → Disetujui | Ditolak`. Use `can("surat.verify")` / `can("surat.approve")` from `src/lib/roles.ts` — never hardcode role checks. `suratActionsFor(status)` returns the correct action buttons.

### Data Layers

Three layers work together:

- **IndexedDB** (`src/lib/idb-store.ts`): in-memory + IndexedDB cache for e-surat records. E-surat store delegates to this.
- **localStorage** (`src/lib/esurat-store.ts`): manages active records, archive, and imported penduduk data. Currently the primary data store in production.
- **Sync Layer** (`src/lib/useSupabaseSync.ts`): write-behind sync — every mutation writes to localStorage first, then attempts Supabase. Graceful fallback. Provides `syncSaveRecord()`, `syncSetStatus()`, `syncArchive()`, `logAudit()`, `healthCheck()`.
- **Supabase**: full relational schema with RLS. DB function `generate_surat_number()` produces sequential numbers in format `0001/S-SKD/470/2026`.

> **Use `isSupabaseConfigured` from `src/lib/supabase.ts` before any Supabase operation.** The sync layer handles this internally.

### Zustand Stores

Four stores manage domain state — all browser-only:

| Store               | Purpose                                              |
| ------------------- | ---------------------------------------------------- |
| `esurat-store.ts`   | E-surat records, archive, attachments                |
| `penduduk-store.ts` | Penduduk data import/lookup                          |
| `settings-store.ts` | Admin settings, warga auth users                     |
| `content-store.ts`  | Berita, pengumuman, agenda (lazy-loaded static data) |

### Template System

`src/data/surat-fields.ts` defines field schemas per letter type (SKD, SKU, SKTM, etc.). Body templates use `{{placeholder}}` syntax — rendered via `renderTemplate()` in `template-store.ts`. `suratActionsFor(status)` and `templateActionsFor(status)` return the correct action buttons for the current status and role.

### Edge Functions (Cloudflare Pages)

All run on Cloudflare Pages (Edge). Service role key and secrets come from Cloudflare Secrets — never exposed to the browser. Shared utilities live in `functions/_lib/utils.ts` (hashOtp, json, corsOptions, base64UrlDecode).

| Function                      | Purpose                                                        |
| ----------------------------- | -------------------------------------------------------------- |
| `api/generate-nomor-surat.ts` | Sequential letter number (`0001/S-SKD/470/2026`)               |
| `api/generate-pdf.ts`         | PDF with QR code. Reads from Supabase; falls back to mock data |
| `api/send-wa.ts`              | WhatsApp via Fonnte. Logs to `audit_log` + `notifications`     |
| `api/verify-surat.ts`         | Public endpoint to verify letter by number                     |
| `api/auth/request-otp.ts`     | Warga login — sends OTP via WhatsApp (SHA-256 hashed)          |
| `api/auth/verify-otp.ts`      | Warga login — verifies OTP hash, creates HMAC-SHA256 session   |
| `api/push/send.ts`            | Send Web Push notifications                                    |

Set Cloudflare secrets before deploying:

```bash
wrangler secret put FONNTE_API_KEY
wrangler secret put SUPABASE_URL
wrangler secret put SUPABASE_SERVICE_ROLE_KEY
wrangler secret put JWT_SECRET   # required in production for warga session signing
wrangler secret put VAPID_PRIVATE_KEY   # for Web Push
wrangler secret put VAPID_PUBLIC_KEY    # for Web Push
wrangler secret put VAPID_SUBJECT       # mailto:admin@serunimumbul.desa.id
```

### Roles & Permissions

`src/lib/roles.ts` defines the role matrix (Super Admin, Operator, Verifikator, Kepala Desa). Always use `can("action")`, `suratActionsFor()`, and `templateActionsFor()` — **never hardcode role checks**.

### PWA

- `public/manifest.json` — PWA manifest with shortcuts; uses PNG icons (icon-192.png, icon-512.png) for broad install prompt support
- `public/sw.js` — Service Worker (offline-first, network-first for API); cache versioned via `?v=__GIT_HASH__` query param — auto-invalidated on every deploy
- Icons in `public/icons/` — both SVG (for SVG-native browsers) and PNG (generated via `sharp` from SVG sources)
- Registered via `useServiceWorker()` hook in `src/routes/__root.tsx` — **only runs in production**, not dev mode
- `vite.config.ts` includes a `buildHashPlugin` that injects the git short hash (or build timestamp) as `VITE_BUILD_HASH` into the client bundle
- SVG favicon in `public/favicon.svg`

## Design System

Tailwind CSS v4 with **CSS-based theming** — no `tailwind.config.js`. All design tokens (colors, fonts, radius, etc.) are defined via `@theme inline {}` in `src/styles.css`. To add a new color or token, edit `src/styles.css` directly.

## Component Utilities

- `cn()` in `src/lib/utils.ts` — clsx + tailwind-merge helper. **Always use this** instead of template literals for conditional Tailwind classes on shadcn/ui components and any component using `cva()`.
- Add new shadcn/ui components with:
  ```bash
  npx shadcn@latest add Button
  ```
  Style is `"new-york"` (configured in `components.json`). All UI components live in `src/components/ui/` with alias `@/components/ui`.

## Database Migrations

Five files in `supabase/migrations/`, run in order:

| File                         | Purpose                                                         |
| ---------------------------- | --------------------------------------------------------------- |
| `001_core_schema.sql`        | Core tables + RLS + `generate_surat_number()`                   |
| `002_warga_auth.sql`         | Warga auth: `otp_requests`, `warga_sessions`, `warga_trackings` |
| `003_attachments.sql`        | Adds `attachments` column to `surat_requests`                   |
| `004_push_subscriptions.sql` | Web Push subscription storage                                   |
| `005_migrate_warga.sql`      | Seeds 5 demo warga into `warga` table                           |
